Clemson Procurement / how it works

A procurement assistant grounded in the actual rules.

Procurement questions turn on exact dollar thresholds, required methods, and approvals. This assistant looks those up in a compiled, human-verified rulebook, shows the reasoning, and cites the policy — rather than generating an answer from similar-looking text. Dollar figures are looked up, not generated.

01

Where a general chatbot falls short on procurement

The everyday tool

The common pattern: attach a PDF, ask a question, get a paragraph back.

For many tasks that works well. For procurement it is risky. These tools work by similarity — they retrieve passages that look related to the question, then a language model writes a fluent answer. In the process, the model can state a dollar threshold that is slightly off, blend a state rule with a federal one, or miss an exception.

Here, "almost right" is not a typo. It can mean a split purchase that should have been bid, a missing quote, or a grant expense that is later disallowed. Because the point of asking is to stay compliant, a plausible but wrong answer is worse than no answer.

What makes it hard
"Three written quotes are required — unless it's on a state contract; but a federal grant imposes its own limit; see also the IT carve-out." — the shape of nearly every real procurement rule

The rules live in long PDFs and sprawling websites, in tables of thresholds, across three overlapping rulebooks — Clemson policy, South Carolina state law, and federal Uniform Guidance for grants — that cross-reference and sometimes contradict each other. Reading is not the hard part. Applying the right rule, and proving you did, is.

02 The core idea

It resolves the governing rule with code and shows the sources, rather than describing the rules from memory.

STEP 01

Understand what you're buying

It works out the three things the rule depends on — roughly how much, what kind of thing (in Clemson's own purchasing categories), and what's paying for it — and shows you, so you can correct it before it answers.

?

Look up the actual rule

STEP 02

Not a similar-sounding paragraph — the governing rule, from a compiled rulebook in which a procurement expert has confirmed each entry. When rules depend on one another, it follows the chain and records each source.

§

Explain it, with sources

STEP 03

It answers in plain language for the reader's role, with a citation on each claim and the reasoning shown. If no rule fits, it says so rather than guessing.

04

What an answer looks like

"Can I buy a $300 table on my NSF grant?"

A regular chatbot returns a paragraph. ask_procurement returns a resolved card: what it understood, the rule it applied, the chain when more than one rule interacts, and a citation on every line.

The dollar figures and methods are looked up — never generated. And because a federal grant is involved, it knows to check the stricter of the state and federal limits, not just one.

Each underlined citation opens the exact policy passage it came from. If the category is wrong, correcting the chip re-resolves the answer.

Had no confirmed rule matched, it would not guess — it would say so and link you to the closest policy and a person who can help.

05

The same answer, shaped for four roles

The same verified rules and the same proof underneath — but the answer is shaped for who's asking. Nothing is reworded into a different meaning; only the depth and the language change.

B

Buyer

"Just don't get me in trouble."

The bottom line, the next two steps, and the guardrails — like a warning not to split an order to dodge a threshold.

O

Business officer

"Did this follow the rules?"

The full obligation, the documents that must exist, and the reimbursement angle — framed for oversight.

P

Procurement pro

"Give me the rule and the cite."

Dense and exact, the full decision chain always shown, every threshold sourced to its section.

H

Help desk & vendors

"Walk me through it."

Step-by-step guidance — with a vendor-facing mode that uses outside language and hides internal-only procedures.

06

Why it's built this way

a. A human signs off on every rule

The system reads the policies and proposes rules, but nothing becomes an answer until a procurement expert confirms it. The AI drafts; a person decides. No extracted dollar figure goes live unreviewed.

b. It shows its reasoning

Every answer carries the chain of rules it applied and a citation for each one, so a buyer can act and an auditor can verify — from the same answer.

c. It refuses rather than guesses

When no confirmed rule clearly applies, it says so and points to the source. In compliance, "I'm not sure — here's where to look" is the right answer, not a failure.

d. It respects overlapping jurisdictions

Clemson, state, and federal rules are modeled as what they are — separate, interacting regimes — and when they conflict on grant money, it applies the stricter one on purpose.

e. It stays current

It re-checks the source websites and documents on a schedule. When a policy changes, the rules built from it are flagged for re-review rather than silently trusted.

f. It's reachable everywhere

The same verified answers come through a web page, by voice, and through a connector other software and AI agents can call — one source of truth, not three.

07

How is it done?

The reliability comes from where the work happens. A doc-chatbot does everything the moment you ask and lets the model generate the number. ask_procurement moves the hard work to build time — a human verifies it, code resolves it — and confines the model to translating between language and structure. The technical core is one boundary:

Stage 1 · LLMGroundparse amount · commodity · funding; classify to a category (correctable)
▸
Stage 2 · pure code · no LLMResolvematch & chain the governing rules → an explained, cited decision path
▸
Stage 3 · LLM, then checkedExplain + verifyphrase for the role · enforce citations · refuse if unsure

The dollar figure lives in Stage 2 — a table lookup, never a generated token. The model never authors a threshold.

Build time — compile policy into a verified model
  1. 1
    Crawl per-jurisdiction allowlists → content-hashed snapshots (change detection) + a captured doc_links graph of hyperlinks & cross-references.
  2. 2
    Extract with Docling's layout model — threshold tables survive as cells, fed to the extractor verbatim; OCR/vision fallback for scans.
  3. 3
    Compile — a schema-first agent emits typed candidate rules {jurisdiction, commodity, funding, amount-range, flags} + a rule_relations graph {excepts, overrides, requires_also, see_also}, each with provenance + confidence.
  4. 4
    Human gate — an SME confirms each draft → immutable, versioned, effective-dated rows in normalized Postgres.

The vector/RAG layer (contextual embeddings + BM25 + rerank) is built only for the open-ended how-to tail — never for rule values.

Query time — resolve, then phrase
  1. 1
    Ground — extract the facts the rules need; classify the commodity to a BuyWays category (shown, correctable).
  2. 2
    Resolve — a pure function over rules + relations: match predicates, walk exceptions/overrides, apply most-restrictive jurisdiction → a cited decision path.
  3. 3
    Generate + verify — phrase in the role's voice; reject unresolvable citations; a verifier confirms the prose matches the decision path and carries no uncited figure, else refuse.

Open-ended "how do I…" questions branch to agentic multi-step retrieval over the narrative layer instead of the resolver.

Empirically, vector-only RAG approaches ~0% on schema-bound threshold lookups and legal RAG tools hallucinate 17–33%; the rule-plus-graph approach reaches ~82% with an auditable trace. The effect is determinism where a wrong number has consequences.
