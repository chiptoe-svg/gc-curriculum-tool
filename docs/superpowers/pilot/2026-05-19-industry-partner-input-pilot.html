Industry Partner Input — pilot ready.

A magic-link survey app that asks the people who hire our graduates what those graduates actually need to know. The data flows back into the career targets the GC curriculum builds toward.

Shipped 2026-05-19 Plan 1 of 3 Tests 122 passing Stack Next.js 15 · Neon Postgres · Resend

01 What this is

The GC Curriculum Tool builds curriculum backward from career targets — the roles our graduates land in. Those targets are the load-bearing definition the whole tool sits on. Until now, the targets were seeded from O*NET and faculty judgment alone. They needed industry voice.

The Industry Partner Input Tool is the upstream feeder. Industry partners receive a personal magic link, land on a short welcome screen, and describe — in their own words — the positions they hire GC graduates into. Job titles, responsibilities, salary ranges, the skills they actually look for, the questions they actually ask in interviews. Faculty sees the raw partner voice. Later in the build, AI will help synthesize themes and propose concrete edits to the curriculum's Know/Understand/Do descriptors — but never auto-write them. Faculty stays in control.

Pilot scope

5–10 trusted industry partners. We watch how they actually use the form — completion rates, where they drop off, what they write into the "none of these fit" escape hatch. The synthesis layer doesn't ship until we have raw input to synthesize from.

02 What's live

Three workflows ship in this pilot.

Faculty

CSV import + invite

Faculty upload a list of partners as a CSV (name, company, email, optional weight and target hints). The tool validates per-row, creates partner records, generates unique magic tokens, and sends invite emails via Resend. Duplicates are skipped. Send failures are surfaced individually.

Partner

Magic-link survey

Partners click their unique link, get a session cookie, land on a welcome screen (first visit) or a personal dashboard (returning visit). They can come back through the same link as often as they want. Inactive partners are revoked with one click.

Partner

Position submission

Three steps: pick the closest career-target match (or write in a label if none fit), describe the position (title, responsibilities, salary, required skills, interview questions, notes), confirm. Drafts auto-create and persist, so partners can step away mid-form and return without losing work.

03 How the pilot runs

End-to-end, from faculty's first action to partner submissions landing in the database.

→ 01

Faculty imports a partner list

A CSV with email, firstName, lastName, company, roleTitle, weight, careerTargetHints. Optional fields can be blank. Faculty hits "Import + send invites." The tool validates each row, creates the partner records, and sends an invite from Resend.

→ 02

Partner receives the invite

Personal greeting, one-sentence ask, single CTA. The link is unique to them — sharing it would let someone else impersonate. Tokens are 192-bit cryptographic randoms; cookies are httpOnly and scoped to the partner route.

→ 03

Welcome → wizard → submission

First visit shows a welcome screen with two cards (describe a position, rate projects — the second is greyed out, deferred to Plan 2). Click into the wizard: pick the closest career target, describe the position, submit. Save-draft is available throughout. Returning visits show the dashboard instead of welcome.

→ 04

Submission lands in partner_submissions

Status flips from draft to submitted, an event is logged. Faculty can read raw submissions immediately via the database; the admin synthesis surface ships in Plan 3.

→ 05

Partner can return any time

Add another position. Resume a draft. Delete a submitted entry if they want to redo it. The same link works for months. I'm done is just a polite "thanks" page — they can still come back.

Faculty UI access

The admin partners page lives at /admin/partners?slug=<PROTOTYPE_SLUG> — same auth pattern as the existing curriculum tool. The slug is the unguessable URL value already in production.

04 What's intentionally not in v1

The pilot is deliberately small. Everything below is on the roadmap — split across two more plans — and was excluded from Plan 1 so we could ship and learn from real partner input first.

Plan 2 · Admin & ratings

Admin operations + project ratings

  • Per-partner edit (name, company, weight, notes)
  • Deactivate / reactivate partners
  • Submissions firehose (all partners, filterable, exportable)
  • Project-relevance ratings (1–5 stars on student projects)
  • Token expiry enforcement
  • Per-partner submission rate limits
Plan 3 · Synthesis

AI synthesis surface

  • Per-target synthesis dashboard for faculty
  • Proposed K/U/D descriptor diffs with rationale
  • Aggregated job titles, responsibility themes, salary distribution
  • Sample partner quotes with attribution
  • Project-rating heat map (courses × projects × weighted avg)
  • Future: cross-target "Karpathy-wiki" synthesis

05 What we learned building it

The form length question is real

A 10-minute form has a non-trivial drop-off rate. We answered this with the "save draft" pattern — partners can leave mid-form and resume cleanly. Drafts auto-create on first edit; the wizard's ensureDraft idempotency pattern prevents double-creates from concurrent clicks. Whether 10 minutes is even the right ceiling will only surface in the pilot.

Anchor on existing taxonomy, but escape-hatch it

The career-target picker asks partners to find the closest match, not their category. The framing matters. If none fit, an explicit "let me describe it" path captures the gap as an unmappedTargetLabel. Those labels are the early-warning signal for career targets we haven't named yet.

Tokens are bearer credentials — treat them as such

The final review caught one real security issue: I'd been writing the magic token into partner_events.opened metadata for debugging. Tokens are bearer credentials — anyone with database read access could have impersonated any partner. Fixed before the pilot opens. The partner is already identified by the partnerId FK; the token adds nothing to the event log.

The auth model is correct, data isolation between partners is airtight, all plan deliverables are present.
— Final code review, 2026-05-19

De-identification vs. weighting in synthesis

The bigger product-design call in the spec wasn't a code question — it was about how the AI synthesis layer should treat partners. Chip pushed back on de-identifying inputs: "Coca-Cola doing something vs. some tiny printer is a signal worth amplifying." The compromise that landed in the design: real identity passed to the LLM alongside an explicit, faculty-set weight column. Weight is the dial; the LLM doesn't get to invent importance from brand recognition alone. That's still ahead in Plan 3.

06 What's next

Immediate (this week)

  • Add Chip + 5–10 trusted partners via CSV import.
  • Send invites. Watch open rates, completion rates, and what shows up in unmappedTargetLabel.
  • Read the raw partner_submissions table directly — no admin synthesis UI yet, but the data is queryable.

Plan 2 — Admin views + project ratings

Bring the admin surface up to operational standard: edit partner records, deactivate, view submissions firehose, export CSV. Add the second contribution surface — project ratings on student work. Wire per-partner rate limits and token expiry. The pilot's lessons inform what UX polish actually matters here.

Plan 3 — Synthesis

The AI synthesis pipeline and per-target dashboard. Faculty reads aggregated insights and proposed K/U/D edits per career target, with the option to scale up to hundreds of partners once the synthesis layer makes that volume legible. This is the surface that closes the loop back into curriculum.

Ready when you are

The tool is live behind the existing GC Curriculum Tool deploy. Open /admin/partners?slug=<slug>, paste a CSV, and the first invites go out. Partner submissions land in the database immediately. The synthesis layer arrives when there's something to synthesize.
