The Problem

Analysis is only as good as the course data underneath it.

Right now the system generates KUD outcomes on-the-fly inside every analysis call — invisible, uneditable, and different every run.

The current prototype produces results like "2 met, 2 missing" for a prereq chain — but the four competencies it evaluates against are the career target's sub-competencies, not the focal course's actual entry requirements. GC 3460 doesn't need "client needs diagnosis" as a prereq; it needs color theory, halftone fluency, and print process literacy. The career target is the wrong frame.

More fundamentally: every faculty member who sees the tool will ask "where did those competency descriptions come from?" If the answer is "the AI generated them inside the analysis call and you can't see or change them," buy-in stops there. The results look like a black box.

The fix is to separate authoring from analysis. A course must have an accepted KUD profile — built with faculty input, editable, versioned — before it can participate in any analysis. The Course Builder is that authoring step.

Design principle: Faculty own what their course claims to develop. The AI drafts; the instructor accepts. Nothing enters the analysis pipeline without a human sign-off.
Tool Ecosystem

Three tools, one pipeline.

The Course Builder is the middle layer — it takes raw course data in and produces approved course profiles that downstream analysis tools consume.

Upstream
Catalog Seed

This tool
Course Builder
Ingestion → profile editing → KUD authoring → faculty approval. Output: a trusted course KUD profile locked in the database.
Downstream

No career target needed for that question. The career target alignment tool remains separate and uses its own frame.
Users

Who uses the Course Builder.

Faculty — primary

The instructor who teaches a course owns its KUD profile. They upload the syllabus and supporting materials, edit the structured profile until it accurately represents what the course does, iterate on the AI-drafted KUD bullets until they feel right, and then accept. Their acceptance is the gate.

The tool should feel like a structured conversation about what their course produces — not a form to fill out or an AI output to rubber-stamp.

Curriculum Coordinator — secondary

Can see all courses and their builder status. Can flag courses as needing attention. Can view but not accept another instructor's KUD profile — acceptance belongs to the course owner. Uses the aggregate view to understand which courses are ready for analysis and which are holding up the coverage matrix.

Workflow

Five stages, one URL per course.

Each stage is a tab on the course builder page at /courses/[code]. Stages are sequential but re-enterable — an instructor can always go back to Stage 3 and regenerate KUDs after editing a project description.

1
Course Info
/courses/[code] → Info tab
The catalog baseline: code, title, credit hours, level (1–4), track, description, prerequisites. Pre-populated from the catalog seed. Each field is editable; fields that differ from the catalog value show a subtle "Edited from catalog" indicator.

This stage is always complete for any course in the database. It does not block progress.
2
Materials
/courses/[code] → Materials tab
File upload for syllabus, assignment sheets, rubrics, past project examples. Each uploaded file is extracted (text or vision for low-text PDFs) and the extracted text is shown in an expandable preview. Instructor can flag a bad extraction ("this scanned poorly — use the typed version instead").

Materials are optional — a course can proceed to the Profile stage with only catalog data — but a course without materials will produce weaker KUD drafts. The Stage 3 prompt shows a notice: "No materials uploaded — KUD draft will rely on catalog description only."
Reuses existing course_materials infrastructure
3
Profile
/courses/[code] → Profile tab
Structured fields that form the evidence base for KUD generation. Three sections:

Learning Objectives — editable list. Pre-populated from catalog if available, or extracted from uploaded syllabus. Each objective is a single sentence in student-can-do form.

Major Projects — editable list. These are the highest-stakes assignments — the ones the AI will treat as the primary evidence for Do-level claims. Order matters: the first project carries the most weight in KUD generation.

Required Incoming Skills — what students need to arrive knowing. This is the course's own statement of its prereq expectations, distinct from the catalog's prerequisites field (which lists course codes, not competencies). The AI uses this field in the gap analysis.

Each section supports free-text editing and item reorder. Changes here do not automatically regenerate KUDs — the instructor decides when to move to Stage 4.
Profile completeness: at least 1 objective + 1 project required to unlock Stage 4
4
KUD Review
/courses/[code] → KUD tab
The core authoring step. See The KUD Iteration Loop for full detail.

The instructor generates a KUD draft from the profile, reviews it, edits as needed (directly or by going back to Stage 3), and iterates until the bullets accurately represent what the course produces. When satisfied, they accept.
Acceptance here is the only gate to Stage 5 and to downstream analysis
5
Approved
/courses/[code] → Status
The accepted KUD profile is visible in a read-only summary. Shows: threshold concept, Know / Understand / Do bullets, acceptance timestamp, and which generation run it came from. A "Revise KUDs" button re-opens Stage 4 (resets to in-progress until re-accepted).

The course now appears as selectable in the Prereq Analysis tool and in the Coverage Matrix. Unapproved courses appear grayed out in those tools with a tooltip: "KUD profile not yet approved — complete the Course Builder to enable this course."
Design Detail

The KUD Iteration Loop

The most important UX design in this tool. The loop is a structured conversation between the instructor and the AI about what the course actually produces.

Three-panel layout

Know

Halftone mechanics: screen frequency, dot gain, tone value increase

Understand

Why ink formulation choices propagate through reproduction quality and cost

Select and justify a Pantone standard, establish reproduction targets, and report results in professional lab format
Conduct controlled ink and substrate testing and interpret results against industry tolerance
Threshold concept

"Color is not a property of the file — it is a property of the ink-substrate interaction. Every reproduction decision is a prediction about a physical process."

Confidence notes

Do bullets grounded in Brand Color Report and Ink Properties Lab — strong evidence. Understand bullets inferred from lab report requirements; medium confidence. No explicit outcomes statement for halftone screening — flagged.

Editing rules

The instructor can edit any KUD bullet directly in the center panel. Direct edits are allowed but the tool nudges them toward making the profile the source of truth: when a bullet is edited directly, a small note appears — "Consider updating the project description in the profile to reflect this — it will make future regenerations more accurate."

Editing anything in the left (profile) panel marks the current KUD draft as stale. The Accept button grays out and a banner appears: "Profile updated — regenerate KUDs before accepting." This prevents accepting a KUD draft that doesn't reflect the current profile.

Why the regenerate nudge, not auto-regenerate: Auto-regeneration on every keystroke is disruptive. An instructor editing a project description mid-thought doesn't want the KUD panel reshuffling. The nudge lets them finish editing the profile, then deliberately trigger a fresh generation when ready.

Iteration flow

  • Instructor reviews the draft and finds a Do bullet is overstated — it reads as a transferable performance but the project is just a procedure-following lab
  • They go back to the left panel and clarify the project description: "follows a set procedure to test ink tack — interpret results but does not design the test"
  • They hit Regenerate. The new draft downgrades that Do bullet to Understand, which is accurate
  • They accept the revised draft

The number of iterations is not bounded. The goal is that the instructor accepts KUDs they stand behind — not a speed run.

Design Detail

Prereq coverage scoring: 1–10, not pass/fail.

Once courses have accepted KUDs, the prereq analysis scores each entry-requirement of the focal course against the prior coursework's accepted KUD bullets. The score is 1–10, not binary.

The focal course's Required Incoming Skills (Stage 3 profile field) becomes the list of competencies to evaluate. For each one, the AI looks at the prior coursework's accepted KUD bullets and asks: how well does this prior work collectively cover this entry requirement?

Example — GC 3460 requires: GC 2070 as prior course
CMYK color theory and halftone mechanics
Entry req from GC 3460 profile
7.5 / 10
GC 2070's accepted KUDs include Know-level halftone exposure via screen printing labs and press measurement work. Covers Know confidently; Understand level is partial — tone value increase is addressed but not formalized. Strong foundation, small gap at the analytical level.
Print process literacy across substrate types
Entry req from GC 3460 profile
5.5 / 10
GC 2070 covers offset and screen printing processes at Know/Understand level. Substrate interaction — the specific frame GC 3460 uses — is not directly addressed. Process literacy is present; substrate compatibility framing is not.
Software file management for production
Entry req from GC 3460 profile
8.5 / 10
GC 2070's accepted KUDs explicitly include Do-level file production workflows and layout documentation. Strong match — this entry requirement is well-covered.

Score interpretation guidance

  • 8–10: Well-covered. The prior coursework develops this entry requirement at the right level with clear evidence. No action needed.
  • 5–7: Partially covered. The prior work addresses the area but not at the depth or in the specific frame the focal course expects. Consider flagging for faculty review or supplemental materials.
  • 2–4: Thin coverage. The prior work touches this area only incidentally. A structural scaffolding gap — may warrant curriculum revision.
  • 1: Not addressed. No prior course in the set develops this entry requirement. A missing scaffold — either the prereq chain is wrong, or the focal course's entry requirement needs revisiting.
Why 1–10 matters: A 5 is actionable in a way "Missing" is not. It says "we're halfway there — one more assignment in GC 2070 covering substrate interaction would close this." A binary result can only say "fix it or don't." The numeric score opens up targeted curriculum adjustment.
Design Detail

Course status model.

Five states, each reflecting how far through the builder a course has progressed.

draft → materials uploaded → profile complete → kuds generated → approved
Status Meaning Downstream eligibility
draft Catalog entry only. No instructor interaction yet. Not selectable in analysis tools. Grayed out with tooltip.
materials uploaded Files uploaded; profile not yet reviewed. Not selectable.
profile complete Objectives, projects, and required skills filled in. KUD generation not yet run. Not selectable.
kuds generated At least one KUD generation run completed. Not yet accepted. Not selectable.
approved Instructor accepted the current KUD draft. Timestamp and IP hash recorded. Selectable in Prereq Analysis and Coverage Matrix.

Editing the profile after approval resets status to kuds generated until the instructor re-accepts. The previous accepted KUD record is retained in history — no data is destroyed.

Design Detail

The approval gate.

How approval status flows into downstream tools.

The course selector component used in the Prereq Analysis tool (and eventually the Coverage Matrix) fetches courses from GET /api/courses. That endpoint will gain an ?approved=true param — when set, it only returns courses with builder_status = 'approved'.

The course selector UI will show all courses but render unapproved ones as grayed-out non-clickable items, each labeled with their current status. This preserves visibility into what's in the catalog while making the gate clear.

Rollout consideration: During the pilot period, before any courses have completed the builder, the prereq analysis tool may need to fall back to showing all courses (including unapproved ones, marked visually). This prevents the tool from being completely empty while the first batch of KUD profiles are authored. This fallback should be removed once a critical mass of core GC courses are approved.
Implementation

Data model additions.

Two changes to the schema: a new course_kuds table and a new status column on courses.

New table: course_kuds

ColumnTypeNotes
course_code PK FK → courses text One active KUD record per course. History kept in course_kud_runs.
threshold_concept text The one idea that, once learned, reorganizes how students see the discipline.
know jsonb string[] 3–5 accepted Know bullets.
understand jsonb string[] 3–5 accepted Understand bullets.
do jsonb string[] 3–5 accepted Do bullets.
manually_edited boolean True if any bullet was edited directly (not just via regeneration). Surfaced to coordinator as a quality note.
source_run_id FK → course_kud_runs uuid Which generation run this accepted version came from.
approved_at timestamptz When the instructor accepted. Null if not yet approved.
approved_by_ip_hash text SHA-256 of instructor IP at acceptance time.
updated_at timestamptz Last modified.

New table: course_kud_runs

Each time the instructor hits "Generate KUDs" or "Regenerate KUDs", a run is recorded. This gives a full history of how the KUD profile evolved.

ColumnTypeNotes
id PK uuid Auto-generated.

model text AI model used.
cost_usd_cents integer

New columnTypeNotes
builder_status text 'draft' | 'materials_uploaded' | 'profile_complete' | 'kuds_generated' | 'approved'. Default: 'draft'. Drives course selector gating.
Implementation

API endpoints.

GET /api/courses/[code]/builder
Returns the full builder state for a course: catalog fields, materials list (with extraction status), profile fields, current KUD draft (if any), approval status, and run history summary. Single fetch to hydrate the entire builder page.
PUT /api/courses/[code]/profile
Updates the profile fields: learning objectives, major projects, required incoming skills. Also updates builder_status to 'profile_complete' when all three fields have content. If course was previously 'approved' or 'kuds_generated', resets to 'profile_complete' and clears approved_at on the course_kuds record.
POST /api/courses/[code]/kuds/generate
Triggers a KUD generation run. Takes no body — reads current profile from DB. Runs the draft-outcomes prompt (course-centric, no career target). Writes result to course_kud_runs. Updates builder_status to 'kuds_generated'. Returns the generated KUD draft plus run ID.
PUT /api/courses/[code]/kuds
Saves the current KUD draft (possibly with instructor edits). Updates the course_kuds record with new bullet values. Sets manually_edited = true if any bullet differs from the source run's output. Does not set approved_at — that requires a separate accept call.
POST /api/courses/[code]/kuds/accept
Marks the current KUD draft as accepted. Sets approved_at = now(), approved_by_ip_hash, and builder_status = 'approved' on the courses record. Returns the updated course_kuds record. Rate-limited per IP to prevent automated approvals.
GET /api/courses?approved=true
Existing courses list endpoint gains an approved param. When true, filters to builder_status = 'approved' only. Used by the course selector in the Prereq Analysis tool. When false or absent, returns all courses (existing behavior preserved).
Implementation

Integration with existing tools.

Course Ingestion (existing)

The existing course_materials table, upload endpoint, and extraction pipeline become the Materials tab (Stage 2). No changes needed to the existing ingestion logic — the Course Builder simply surfaces it within the unified workflow. The builder_status advances to 'materials_uploaded' when the first file extraction completes successfully.

Course Profiles (existing)

The course_profiles table currently stores an AI-synthesized narrative profile (summary, competencies). This continues to exist for the course profile synthesis workflow. The new course_kuds table is separate — KUD authoring is a distinct step from profile synthesis, though both draw from the same materials and profile fields.

Prereq Analysis (prototype)

The prototype analysis tab currently generates KUDs on-the-fly inside the analysis call with a career target as the frame. Once the Course Builder is live, the prereq analysis is redesigned to:

  • Only show approved courses in the selector
  • Read accepted KUDs from course_kuds rather than generating them
  • Score using the focal course's Required Incoming Skills as the evaluation frame (not career target sub-competencies)
  • Output 1–10 scores per entry requirement with cited reasoning
  • Remove the career target selector from the prereq tab entirely

Coverage Matrix (future)

The coverage matrix — which shows how the full 120-credit program covers each career target's sub-competencies — will be able to use approved course KUDs as an additional input alongside coverage scores. A course with strong "Do-level" KUD outcomes on print production contributes to the Production & Ops coverage in a way that can be quantified and compared across the program.
