GC Curriculum Tool

The implementation design for Builds 1–3 (Foundation, Course Content, Coverage Analysis) from the source GC Curriculum Design Tool specification.
Date 2026-05-17 Status Draft for user review Scope Builds 1–3 (of 7)

This document is the implementation design for the first three builds of the GC Curriculum Tool. It resolves the architectural and scope questions the source spec leaves open and produces a buildable plan.

Builds 4–7 (Proposals, Curriculum Map, Sankey/Sequence, Resource Summary, Presentation View) and the full Assessment Framework are out of scope for this design. Each will get its own design document once v1 is real with seeded data.

Why not the source spec's "Claude artifact"

The source spec hedges between artifact-based prototype and full app. v1 has four user roles eventually, real persistence, real AI calls, and PDF export. Claude artifacts are not the right home for any of these. Starting on Next.js avoids a costly migration at Build 4.

Why not fork DCM or Skills-Mapping

DCM (WolfgangFahl/dcm) is a research artifact (5 stars, 1 fork) in Python + NiceGUI; its competency-tree paradigm conflicts with the spec's flatter target → sub-competency structure; it has no AI, no auth, no Proposal concept — the hard parts of this tool.

Skills-Mapping (Humanitariansai/Skills-Mapping) is currently an empty repo (all data/code directories empty as of 2026-05-17). The genuine insight from that line of research — help advisors set up career targets, don't make them stare at blank fields — is folded in via O*NET KSA fetching and the AI competency-unpacking endpoint.

§2Data Model

Drizzle schema. Only the entities needed for Builds 1–3 are included. Proposal-related and assessment-related entities are deferred to their respective design docs.

Entities

// Users — single admin row seeded; multi-user comes at Build 4
users {
  id: uuid (pk)
  name, email (unique)
  role: enum('admin', 'faculty', 'panel_member', 'viewer')   // only 'admin' used in v1
  created_at
}

// Courses — identity record
courses {
  id, code (unique), title, credit_hours, level,
  delivery_format: enum('in_person', 'hybrid', 'online'),
  owner_id → users.id (nullable; null in v1),
  official_record_id → course_records.id (nullable),
  retired: boolean, retired_at, created_at
}

// Prerequisites — many-to-many self-reference
course_prerequisites { course_id, prereq_course_id }

// Course records — versioned content
course_records {
  id, course_id → courses.id,
  description: text,
  know_outcomes: jsonb,         // string[]
  understand_outcomes: jsonb,   // string[]
  do_outcomes: jsonb,           // string[]
  syllabus_text: text,
  is_official: boolean,
  marked_accurate_by → users.id, marked_accurate_at,
  notes: text,
  created_at
}

// Projects — separate table, not embedded
projects {
  id, course_record_id → course_records.id,
  name, description,
  competency_tags: jsonb,       // string[] of sub_competency ids
  display_order
}

// Career Targets
career_targets {
  id, name, short_definition,
  industry_contexts: jsonb,
  know_descriptors, understand_descriptors, do_descriptors: jsonb,
  panel_notes, panel_members: jsonb, defensibility_note,
  last_reviewed_at,
  soc_code: text,               // "13-1161.00" — anchors O*NET fetch
  soc_label: text,              // cached SOC title
  display_order
}

// Sub-competencies — children of career targets
sub_competencies {
  id, career_target_id → career_targets.id,
  name, know_descriptor, understand_descriptor, do_descriptor,
  display_order
}

// Coverage scores — AI-generated, versioned
coverage_scores {
  id, course_record_id, sub_competency_id,
  kud_level: enum('know', 'understand', 'do', 'not_addressed'),
  confidence: enum('high', 'medium', 'low'),
  ai_reasoning: text,           // 1–3 sentences, required
  disputed: boolean, dispute_note: text,
  scored_at,
  ai_provider: text, ai_model: text   // for reproducibility
}

// O*NET response cache
onet_cache { soc_code (pk), payload: jsonb, fetched_at }

// Prerequisite competencies — what students should walk INTO this course with
// (distinct from course_prerequisites, which is registrar course-to-course)
prerequisite_competencies {
  id, course_record_id → course_records.id,
  sub_competency_id → sub_competencies.id,
  expected_kud_level: enum('know', 'understand', 'do'),
  rationale: text,            // why this course needs it
  source: enum('faculty', 'ai_suggested', 'ai_confirmed'),
  display_order
}

Notable design calls

  • Projects in a separate table, not embedded JSON. Costs nothing in complexity; lets us query "all projects tagged with sub-competency X" later (useful for Build 4 Impact View).
  • ai_provider and ai_model recorded on every coverage score. Not in the source spec. Reproducibility matters: when you switch providers or models later, you'll want to know which scores came from which.
  • Disputes preserved across reruns. When a course record's coverage is rescored, the new coverage_scores rows inherit disputed=true and dispute_note from the prior row for the same (course_record × sub_competency) pair. Faculty pushback should not silently vanish.
  • At most one official record per course. Enforced by a partial unique index on course_records(course_id) WHERE is_official = true. A newly-created course has zero official records until "Mark as accurate" runs the first time.
  • Prerequisite competencies are the inverse of coverage. Coverage answers "what does this course build toward?"; prerequisite competencies answer "what does this course expect students to walk in with?" The combination lets us detect scaffolding failures — when a downstream course's expectations exceed what any upstream course actually develops. Distinct from the registrar's course_prerequisites (just "must have taken X before"); this is the competency-grain version. Manning's Learning Progressions (D7) and Scope and Sequence (D16) skills encode the reasoning framework.

The Build 2 → Build 3 loop

Where the AI value shows up:

  1. Open /courses/GC-3460/edit (identity already seeded).
  2. Paste syllabus text into the syllabus field.
  3. Click "Draft outcomes from syllabus" → AI populates Know / Understand / Do bullets and a draft description.
  4. Edit outcomes; add 2–4 projects with descriptions; tag each project with sub-competencies.
  5. Click "Mark as accurate" → transaction flips is_official, creates new record, enqueues 5 background scoring calls (one per career target).
  6. Course page updates as scores arrive. Each coverage row shows KUD pill + confidence badge + click-to-expand reasoning.
  7. If a score is wrong, click "Dispute", write a note. Note preserved across reruns.
  8. /coverage heat map reflects the new scores.

Key design calls

  • Coverage analysis runs per-target, not per-sub-competency. One call returns scores for all sub-competencies under one target. Cheaper, more coherent reasoning, cleaner UI.
  • "Mark as accurate" is the analysis trigger, not "Save". Drafts autosave without invoking AI; scoring runs only at accuracy mark.
  • No Proposal mode in v1. Course page shows Official Record state only. "Start a Proposal" is a disabled button with a "Coming in Build 4" tooltip.
  • Heat map is the only visualization in v1. Sankey, Sequence Map, and Gap Panel are Build 5–6.
  • Prerequisite Gaps appear as a list, not a diagram in v1. A faculty-resonant section on the Course Page: "These competencies are expected when students arrive, but no upstream course develops them to the expected level." The Sequence Map (Build 6) will eventually overlay this on the prerequisite network with red arrows for failed scaffolding chains, but the list alone is what faculty react to.

§4AI Integration & Manning Skills

Endpoints

EndpointTriggerInputOutput
POST /api/ai/draft-outcomes Course edit page syllabus text + target descriptors {description, know[], understand[], do[]}
POST /api/ai/score-coverage Mark as accurate (per target) course record + one career target [{sub_competency_id, kud_level, confidence, reasoning}]
POST /api/ai/unpack-competency new Sub-competency "Help draft KUD" button sub-competency name + target context {know_descriptor, understand_descriptor, do_descriptor}
POST /api/ai/suggest-prerequisites new Course edit "Suggest prerequisite competencies" course record + all sub_competencies [{sub_competency_id, expected_kud_level, rationale}]
POST /api/ai/analyze-prerequisite-gaps new After Mark as accurate (background) course record + its prereq competencies + upstream coverage [{prereq_competency_id, status: 'met'|'underdeveloped'|'missing', upstream_evidence, reasoning}]
POST /api/ai/rescore-all Target updated (admin) batch, background rescore every course's coverage for that target
POST /api/onet/fetch-ksas new Career Target "Suggest from O*NET" SOC code structured KSAs (cached 24h)

The O*NET endpoint is not an AI call — it's a direct API proxy with caching.

Manning skill encoding strategy

The source spec mandates encoding four Manning domain skill sets (Domains 7, 16, 13, 9) into AI system prompts at build time. Approach:

  1. At implementation start, fetch github.com/GarethManning/education-agent-skills and read the SKILL.md files.
  2. Encode each skill's reasoning framework into the relevant prompt file. Where the spec chains two skills, the prompt walks both frameworks in sequence.
  3. If the repo is unavailable, fall back to the source spec's own summary descriptions of each Manning skill.

Prompt files live as version-controlled markdown with YAML frontmatter:

/lib/ai/prompts/
  draft-outcomes.md             uses: Backwards Design + KUD Chart Authoring +
                                      Threshold Concept Translation
  score-coverage.md             uses: Coverage Audit + KUD Chart Authoring +
                                      Assessment Validity + Developmental Band Translation +
                                      Disciplinary AI Reliability
  unpack-competency.md          uses: Competency Unpacking
  suggest-prerequisites.md      uses: Learning Progressions + Scope and Sequence +
                                      Backwards Design
  analyze-prerequisite-gaps.md  uses: Learning Progressions + Scope and Sequence +
                                      Developmental Band Translation
  shared/kud-rubric.md          the KUD scoring rubric
  shared/career-target-frame.md   how to reason about target sub-competencies
The "AI reasons out loud" rule, mechanically enforced

Every coverage score includes a reasoning field of 1–3 sentences citing specific outcomes or projects. The JSON schema rejects empty or trivially short reasoning; the call is retried with a clarifying message. This matters because the source spec's stance — "the tool does not present AI output as authoritative fact" — depends on it.

§5Build Sequence & Milestones

Each milestone is independently usable and ends with a real-data gate. Don't start the next milestone until the prior one is real.

M-trial — Faculty-facing prototype
~6–7 days

Purpose: A standalone demo page faculty can actually use on their own. Validates AI prompts against real syllabi before architectural commitment, and gives early faculty something concrete to react to.

Page structure (single route, /)

  1. Hero + bigger-picture intro. Plain-language: what the full tool will do, why it matters, what this prototype shows and doesn't. ~200–300 words.
  2. Instructions — short numbered list:
    • Paste an upstream course's syllabus (or "Load GC 3460 example")
    • Paste a downstream course's syllabus (or "Load GC 4060 example")
    • Pick a career target
    • Click "Analyze"
    • Click any AI reasoning to expand it; flag anything that looks wrong
  3. Interactive form — two syllabus textareas with example-loader buttons, career target dropdown, Analyze button.
  4. Output area:
    • Two side-by-side KUD outcome cards (AI-drafted, clearly shown as drafts)
    • Heat map: 2 rows × N columns by KUD level, click cell → reasoning + Flag button
    • Prerequisite Gap Analysis panel: status + reasoning + Flag button per gap
  5. Footer: "This is a prototype. The full tool ships in ~3 months. Feedback: <email>."

Technical components

  • Next.js page at / with API routes for AI calls
  • AI provider abstraction carries forward
  • Manning-skill-encoded prompts carries forward
  • Seed data (5 targets + sub-competencies + KUD descriptors) carries forward
  • Heat map + prerequisite gap components carries forward
  • prototype_runs + prototype_flags log tables for prompt-tuning analysis
  • Cost protection: per-IP rate limit (10/hour), daily cap ($5), email alert if hit
  • Unguessable URL slug: gc-curriculum-tool.vercel.app/preview/<slug>

Access model

Unguessable URL, no password. Faculty get the link from you directly. If you want a password later (e.g., before sending to industry panel), it's a one-line change.

Sample syllabi

Pre-loaded from the source spec for GC 3460, GC 4060, GC 4070, GC 4400, GC 3720, GC 3400 via "Load example" buttons. Faculty can replace with their own.

Gate

A faculty member you've never demoed it to can open the link, follow the instructions, paste their own syllabus, get a substantive result, and flag at least one piece of AI reasoning. You can read the flags afterward and use them to tune prompts before M0.

~80–85% of code carries forward

AI provider, prompts, seed data, heat map, gap component, JSON schemas all become foundation for M1–M3. The throwaway: single-page form UI, intro copy, "Load example" buttons (~15–20% of total prototype code).

M1 — Foundation
~3–4 days · maps to Build 1
  • Drizzle schema for: users, courses, course_prerequisites, career_targets, sub_competencies (no course_records yet — identity only).
  • Seed script (scripts/seed.ts) parses the source spec markdown to populate:
    • All 5 career targets with seed KUD descriptors, industry contexts, defensibility notes, SOC codes.
    • Seed sub-competencies (typically 5–7 per target) inferred from the spec's "Core competency areas" lists.
    • All 19 courses (13 with data + 6 without) as identity records.
  • /targets index + /targets/[id] editor: KUD descriptor editing, sub-competency CRUD with drag-reorder, panel info, defensibility note.
  • /courses index + /courses/[id] (placeholder view) + /courses/new + /courses/[id]/edit (identity fields only).
  • POST /api/onet/fetch-ksas + "Suggest from O*NET" button. KSAs presented as a checklist.
  • POST /api/ai/unpack-competency + "Help draft KUD descriptors" button on sub-competency editor.
Gate

All 5 career targets editable with full KUD descriptors and sub-competencies. The three SOC-anchored targets Account Mgmt Brand Strategy Prod & Ops have at least one panel-selected KSA from O*NET. At least one sub-competency was drafted with AI assistance. All 19 courses listed and editable.

M2 — Course content + AI drafting
~5–6 days · maps to Build 2
  • Drizzle schema additions: course_records, projects, prerequisite_competencies. Partial unique index for is_official.
  • Course page (Official Record view mode) renders the spec's full layout. Coverage and Prerequisite Gaps sections are placeholders ("full analysis lights up in M3") until M3.
  • Course edit page: description, KUD outcome editors, syllabus text area, project editor with competency tag picker, prerequisite competency editor with AI-suggest button.
  • POST /api/ai/draft-outcomes wired to "Draft outcomes from syllabus" button.
  • POST /api/ai/suggest-prerequisites wired to "Suggest prerequisite competencies" button.
  • "Mark as accurate" action: transaction that flips is_official and timestamps. Analysis enqueues stubbed (return OK).
  • Seed script extension: drafts initial outcomes and prerequisite competencies for the 13 courses-with-data from the spec's prose, stored as draft records for review.
Gate

You've marked 3 courses accurate with real outcomes, projects, and prerequisite competencies. AI drafting and prerequisite suggestion noticeably reduce work vs. typing from scratch. The Course Page reads clearly enough that a faculty member could understand it.

M3 — Coverage + prerequisite gaps + heat map
~6–7 days · maps to Build 3
  • Manning-skill-encoded prompt files written and reviewed: score-coverage.md, analyze-prerequisite-gaps.md, shared rubric and frame files.
  • POST /api/ai/score-coverage with strict JSON schema.
  • POST /api/ai/analyze-prerequisite-gaps — transitively walks course_prerequisites to assemble upstream coverage, evaluates each prerequisite competency against what's actually developed upstream.
  • "Mark as accurate" enqueues background work in order: (1) 5 coverage scoring calls, one per target; (2) 1 prerequisite gap analysis call once coverage is in.
  • Dispute preservation across reruns for both coverage and gap analysis.
  • Course Page: coverage rows grouped by target with click-to-expand reasoning + dispute UI. Prerequisite Gaps section lists expected-but-undeveloped competencies with upstream evidence and reasoning.
  • /coverage heat map: courses × sub-competencies, color-coded by KUD level. Click cell → side panel.
  • Admin "Rerun analysis" button per course.
  • POST /api/ai/rescore-all on career target update; also re-triggers downstream prerequisite gap analyses where chain coverage may have shifted.
Gate

Coverage scores running on 5+ courses against all 5 targets. Prerequisite gaps surfaced on at least one course where a real gap exists — the GC 4060 → GC 3460 chain is the strongest candidate from current data. At least one disputed score with a note. The heat map and gap view are presentable to the industry/faculty panel without apologizing for them.

v1 total estimate

~4.5 weeks of focused work for one developer including the prototype: M-trial 6–7 days + M0 ½ day + M1 3–4 days + M2 4–5 days + M3 3–4 days. The M2 and M3 estimates shrink (vs. the no-prototype path) because the AI prompts, seed data, heat map, and prereq-gap components are already proven by then. The faculty trial during the prototype also reduces the risk of late schema or rubric changes.

Going without the prototype would cost ~3.5 weeks of focused work, but with materially higher risk on M3 quality and faculty acceptance.

§6Open Questions for User Review

These don't block writing the implementation plan but should be answered before M0 starts.

  1. Domain / hosting name. Vercel deploys to *.vercel.app by default. Do you want a custom domain (e.g., curriculum.clemson-gc.edu) and is Clemson IT involved in DNS?
  2. GitHub repo. Private repo created at github.com/chiptoe-svg/gc-curriculum-tool.
  3. OpenAI API key. Do you have an API key with credit, or do we need to budget for the initial $20 setup at the OpenAI console?
  4. O*NET API key. Free but requires registration. Either register one in your name with a project email, or sign up directly at services.onetcenter.org/developer/signup.
  5. xAPI integration (future). DCM uses xAPI statements for learner achievement data. Worth keeping in mind as the wire format for future Build 4+ assessment data flowing in. Not built in v1; just noted.
