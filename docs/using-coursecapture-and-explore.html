Faculty Guide

Using CourseCapture & Explore

A practical walkthrough of the per-course audit workflow that produces a Course Outcome Profile, and the Explore module that turns that snapshot into prescriptive alignment analysis and what-if scenarios.

Audience GC faculty using the tool on their own courses Time per course 20–45 minutes for a first capture Date 2026-06-03

§ 1Overview

The two tools work as a pair. CourseCapture is descriptive: it produces an evidence-backed Course Outcome Profile that describes what the course is evidenced to require, assess, and support — competencies with K/U/D depth ratings, evidence excerpts from materials, and audit notes about prerequisites, objective alignment, and cross-source conflicts. Explore is prescriptive: given a confirmed snapshot, it compares the course to a target (instructor-defined goals, or what downstream courses expect from incoming students), surfaces gaps, and proposes ranked recommendations. Explore also supports what-if scenarios: propose a change to the course and see how the alignment numbers would shift.

Both tools are per-course. You open them through the same session, on the same course code. CourseCapture lives at /capture/<course-code>; Explore at /explore/<course-code>. Each course's page links to the other (top-right of the header).

Read in this order

You only need to read sections about Explore (§10–14) once you have at least one confirmed snapshot for a course. Explore refuses to open if there's no snapshot — it sends you back to CourseCapture first. Sections §15–16 cover the /courses roster and the prerequisite-gap view — useful once several courses have been captured.

§ 2How you reach the tool

The tool lives on two surfaces that serve different purposes. Understanding which URL to use for which task will save you a frustrating moment the first time a browser blocks the microphone.

The public landing — browse without logging in

Point any browser on the Clemson LAN (or VPN) to http://130.127.162.180:3000/. No username, no password. You land on a course listing that shows every course in the catalog with its current capture status and "by [instructor]" attribution. Each row has a View link that opens the read-only profile for that course — useful for quickly checking whether a course has been captured and what it says, or for sharing a profile link with a chair, colleague, or accreditor without giving anyone edit access. View links stay on HTTP and require no authentication.

Edit pages — HTTPS via Tailscale, Basic Auth once per browser

Edit links go to https://admins-mac-studio-2.tailb723c1.ts.net/capture/<code>?slug=…. This address uses Tailscale Funnel, which means it's a proper HTTPS secure context reachable from anywhere — off-campus, over VPN, or from a phone. The first time your browser visits this host it will prompt for a username and password (HTTP Basic Auth). Enter the credentials your administrator provided; the browser remembers them for subsequent visits, so this is a one-time prompt per browser. Edit pages cover CourseCapture, Explore, Program, Admin, and Settings — anything that writes data.

Why the split matters for the microphone

Browsers enforce a rule: microphone access is only permitted on secure contexts (HTTPS). The http://130.127.162.180… address is plain HTTP, so the mic button on that surface will always be blocked by the browser. If you want to use voice input during the audit chat, use the edit-surface URL (https://admins-mac-studio-2…) — the mic works there because it's a top-level HTTPS context. This replaces an earlier approach that tried to proxy mic access through an iframe, which browser security rules ultimately blocked.

Bookmark the edit address

The slug is part of the edit URL — keep that link handy. If you lose it, ask your administrator to regenerate it or look it up on the Admin page. The public landing (http://130.127.162.180:3000/) never needs a slug; the edit surface always does.

§ 3Before you start

Access

The tool is slug-gated: you open it through the access URL your administrator shared. The slug is part of the URL (?slug=…). All pages — CourseCapture, Explore, Program, Settings — pass the same slug forward; just keep the tab open and the navigation will preserve it.

The course must exist in the catalog

The 120-course GC catalog is seeded into the tool. If you're capturing a course that isn't in the catalog (a new course, a renamed special-topics section), contact the administrator before you start — the capture page expects a catalog row to attach the snapshot to.

Have your materials handy

The richer the materials the auditor can read, the less work the audit chat has to do. Three categories of inputs the system can ingest without you uploading anything:

  • Catalog values (description, prerequisites, objectives, projects, required incoming skills) — sourced from the shared GC course sheet; click Sync from Sheet on the Materials panel to pull the latest.
  • Canvas content (syllabus body, assignments + rubrics + point values, modules, pages, discussions, quizzes — Classic and New Quizzes, file metadata) — connect Canvas once per course; the capture page imports on demand.
  • Linked content referenced from any of the above — Google Docs, Sheets, Slides (must be shared "Anyone with the link"), and Drive PDFs are fetched automatically when their URLs appear in materials.

Materials you do upload yourself live alongside the rest: rubrics-as-PDFs, project briefs, lab handouts, anything the auditor needs to read that isn't in Canvas or behind the catalog row.

§ 4CourseCapture · Loading materials

Open /capture/<course-code>. The page splits into a Materials & catalog panel (top) and the audit chat (below). Spend a couple of minutes on the materials panel before talking to the auditor — every item you load is evidence the auditor can read directly instead of asking you to recall.

The Materials panel, item by item

1
Sync from Sheet. The top of the panel shows the catalog values currently in the system (description, prereqs, objectives, projects, incoming skills). If you've edited your course's tab on the shared sheet since the last sync, click Sync from Sheet to pull the latest. The button only touches this course's row.
2
Connect & Import from Canvas. The first time, you'll paste a Canvas access token and the course's Canvas course ID; that pair is stored per-course. Subsequent visits just need an Import from Canvas click. The import pulls the syllabus body, every assignment with its rubric and point value, every page, every discussion topic, every quiz (both Classic and New Quizzes APIs), and file metadata. Each item appears as a row in the Materials list with its kind and source labeled.

The dialog has a Skip unpublished items checkbox that's on by default. Most Canvas courses carry a graveyard of half-built draft assignments, pages, and quizzes that the auditor would otherwise read as real coverage. With the box checked, those drafts are filtered server-side; the import summary tells you how many were skipped. Uncheck if you specifically want the drafts included — they'll arrive tagged [unpublished] in the imported text so the auditor knows what it's looking at.
3
Upload anything else. Drag-and-drop rubrics, project briefs, lab handouts, exam keys, syllabus PDFs. Supported: PDF, DOCX, PPTX, XLSX, CSV, HTML, images (PNG/JPG, OCR'd), TXT, MD. Legacy Office formats (.doc, .ppt, .xls) are auto-converted via LibreOffice headless on upload. The auditor reads the extracted text and uses it as direct evidence.
4
Linked Google/Drive content is fetched automatically. If any of your loaded materials reference a Google Doc, Sheet, or Slides URL, or a Drive PDF link, the system follows the link and fetches the content — provided the file is shared with "Anyone with the link" view access. Each fetched file shows up as its own row in Materials, tagged with what it is. The audit chat will treat the fetched content as readable evidence; if a link can't be fetched (permissions, deleted file), it appears as an inaccessible reference instead.
5
Re-extract Canvas files (optional). If the extraction pipeline has been upgraded since your last import — for example, after the LAN deploy turned on figure descriptions, or after legacy Office support was added — click Re-extract Canvas files on the Materials panel. Paste your Canvas token when prompted; every Canvas file attachment for this course is re-run through the current extractor in place, so existing rows pick up the better text without you having to re-import.
Heads up — content the system can't read

The Canvas import pulls the syllabus body, assignment descriptions and rubrics, module item titles, Canvas Pages content, Canvas file attachments (PDF/DOCX/PPTX/XLSX/CSV/HTML/image are extracted server-side; legacy Office formats are auto-converted first), discussion prompts and graded-status flags, and quiz content from both Classic and New Quizzes (questions, answer choices, points, correct-answer marks). It also pulls YouTube captions when available for linked videos. By default, unpublished assignments, pages, discussions, quizzes, modules, and module items are skipped (see step 2 above); if you choose to include them, they arrive tagged [unpublished] so the auditor scores them accordingly. The import does not read: video content on platforms other than YouTube (Vimeo, Panopto, Canvas Studio, Loom, Kaltura), discussion replies, or external LTI tool content. The auditor will notice references to these in the materials it can read and surface each one with its title or URL — but the scoring will stay conservative until you tell it what's in the inaccessible content. The full source-coverage table is in the Background document.

Figure descriptions on the LAN deploy

The PDF extractor (Docling) is paired with a local vision-language model that writes a one- or two-sentence description for each chart, diagram, or figure into the extracted markdown. This closes a real evidence gap on lecture slides and research papers, where meaning often lives in the figure rather than the prose.

What "good enough" looks like at this stage

You don't need every possible material loaded before starting the audit chat. The auditor will surface what's missing as the conversation progresses. A reasonable starting set:

  • Catalog row up to date (Sync from Sheet just before starting).
  • Canvas imported if the course uses Canvas substantively.
  • Major project briefs and rubrics uploaded (the highest-stakes assignments).
  • The syllabus PDF uploaded if it's not in Canvas.

How material compression works

The auditor's prompt has a hard upper size limit. For courses with many or lengthy materials, keeping every file verbatim would push past that limit and drop some materials from the auditor's view entirely. To avoid that, the system automatically generates a structured summary of long reference-style materials at upload time — so the auditor retains a faithful view of every piece of content even for the largest courses.

What gets summarised

Compression applies to materials that exceed roughly 15,000 tokens: textbook PDFs, reading PDFs fetched from Drive, YouTube transcripts, and plain PDF or DOCX uploads. These tend to be reference-dense and compress well without losing the signal the auditor needs.

What is always kept verbatim

Canvas pages, the syllabus, assignments, modules, quizzes, discussions, and Google Docs/Slides/Sheets are never summarised — the auditor needs the exact assignment language, rubric criteria, and point values to score competency evidence accurately. These items carry the assignment-grade detail the audit depends on.

How it looks in the Materials list

Each row in the Materials list shows an estimated token count. Rows at or above 20,000 tokens are marked in amber; rows at or above 50,000 are marked in red. For a material that has been summarised, the row displays a summary (~Xk tok) chip alongside an audit sends ~Xk tok note so you can see how much the prompt footprint was reduced. A per-row summarize checkbox lets you fall back to the full verbatim text for any specific material when you need the auditor to read every word — for example, if a course reader contains assessment-critical rubric language embedded in prose.

The total-prompt indicator

The Materials & catalog panel header shows a running total of the audit prompt: Audit prompt: ~Xk from materials. This is the sum of what the auditor will actually see, accounting for summaries on rows where summarize is on. The total is colored:

  • Grey — under 150k tokens. Plenty of headroom.
  • Amber — 150k to 220k tokens. Getting large. The indicator suggests running Compress existing materials to shrink it.
  • Red — 220k tokens or more. Approaching the audit-prompt cap (272k). At this point the prompt may be rejected by the model. The indicator advises compressing reference materials and then ignoring uploads you don't need (an ignore checkbox sits next to each row's summarize).

The thresholds are conservative — the cap is 272k tokens on the smaller model and larger on others, but staying well under gives the auditor room for the conversation transcript that builds up over the session.

If the total is too high

Two levers, in order of preference:

  1. Compress existing materials. One click. Eligible long reference rows get a summary; the audit prompt switches to using those summaries automatically. This is reversible per-row via the summarize checkbox if you ever need verbatim text on a specific material.
  2. Ignore unused uploads. The ignore checkbox on a row keeps the material in the database but excludes it from the audit prompt entirely. Use this for materials that don't actually belong to this course (e.g., a stray file from another section that landed here by mistake).
  3. Ignore specific Canvas items. Canvas-list materials (Assignments, Discussions, Quizzes, Pages, Module List) carry a disclosure beneath the row showing every item the auditor will see. Click the ▸ N items — click to manage link to expand the checklist; uncheck any item to drop it from the audit while leaving the rest of the material active. Useful when one specific assignment is irrelevant or misleading but the other ten on the list are fine. Re-check to bring it back.

Deleting a material is fine too, but irreversible — it removes the row and the blob storage. Ignore is the safer first move.

Backfilling materials uploaded before compression shipped

Materials already in the system before this feature was deployed won't have summaries yet. Click Compress existing materials once per course to generate summaries for those rows in place. New uploads are summarised automatically on the way in.

§ 5CourseCapture · Identifying yourself as the auditor

Because the same course code can be taught very differently by different faculty members, depth scores are now recorded per instructor. A GC 3550 taught by one person with a heavy press-lab emphasis produces a different profile than a GC 3550 taught another way — averaging those together would erase information worth preserving. The instructor identity is captured at the start of every session and stamped on every snapshot and audit message.

The session-start chooser

When you open /capture/<course-code>, a modal appears before the audit panel loads. It presents two choices:

  • Instructor dropdown. Select your name from the list of faculty associated with this course. If your name isn't there, ask the administrator to add it — the list comes from the course's instructor roster in the system.
  • Fresh start vs. Build on prior session. Fresh start opens a clean audit chat with no prior transcript. Build on prior session loads the most recent audit session for this instructor on this course, so you can pick up where you left off without replaying the whole conversation. Choose Build on prior session when you're returning to a course you've already partially audited; choose Fresh start for a new semester's capture or a major revision where the prior session's context would mislead the auditor.

Confirm your choices and the audit panel loads.

The Auditor badge

Once the session is open, an Auditor: [your name] · change badge appears at the top of the audit panel. It stays visible throughout the session so you always know whose lens the auditor is using. Click change to return to the chooser — for example, if you're an administrator stepping through multiple instructors' captures in sequence, or if you opened the wrong name by mistake. Changing the auditor identity mid-session starts a fresh chat for the new instructor; the prior session's messages are preserved under the previous instructor's name.

Why this matters for scoring

Scores in the Program Coverage Matrix and in Explore reflect whose profile is the most recent snapshot. If two instructors have both captured GC 3550, the matrix shows both rows separately — one per instructor — rather than merging them. When a downstream course asks "does GC 3550 cover X?", the answer can differ by instructor. The split is intentional: it surfaces real variation instead of hiding it in an average.

§ 6CourseCapture · The audit chat

The chat panel sits below the Materials & catalog panel. The auditor opens by reading everything in the materials and writing three short paragraphs:

  1. A one-sentence summary of what the materials show overall.
  2. A finding — the single most consequential gap, contradiction, or missing piece, with specific evidence cited (assignment name, rubric criterion, objective number).
  3. One focused question that follows up on that finding.

From there it's a conversation: you reply (typing or voice), the auditor acknowledges what you said and asks one new question. The discipline is deliberate — one question per turn, finding-and-question always on the same topic, no batched lists.

What the auditor is checking

Across the conversation, the auditor systematically covers seven audit areas. You don't need to drive these in order — the auditor weaves them in based on what the materials surface — but it helps to know what's coming:

Audit areaWhat it's probing
Prerequisite sufficiencyFor each catalog-listed required incoming skill: does an assignment require it, and do students actually arrive with it? If a prereq course has its own captured profile, the auditor cites that profile directly instead of asking you to recall.
Objectives vs. evidenceTwo directions. Objectives without material evidence ("syllabus says students will analyze X but no assignment requires analysis"). Material outcomes not captured in objectives ("the capstone teaches Z, which isn't in the listed objectives").
Cross-source conflictsPoint values that disagree between syllabus and Canvas, assignments listed in one source but not the other, outcomes language that drifts. The auditor asks which source is current.
Bloom's-level probeFor the highest-stakes assignments, what cognitive level students actually operate at — remembering vs. applying vs. analyzing vs. creating — and what evidence supports the classification.
Threshold conceptThe one conceptual shift that separates students who truly get this course from students who memorized it. What students consistently misunderstand before the shift happens.
Foundational dispositionsFor each of the five baselines — Agency, Attention to Detail, Resilience, Curiosity, Communication — what conditions the course creates (or doesn't) to develop the disposition. Develop, not teach — dispositions are demonstrated through behavior, so the question is whether assignments demand the behavior.
Inaccessible referencesVideos on platforms other than YouTube (Vimeo, Panopto, Canvas Studio, Kaltura, Loom), external LTI tool content, and file attachments in formats the extractor can't read — named in the materials but not extractable. The auditor surfaces each with its title or URL and asks what's in it, so scoring isn't blind.

Voice input

The microphone icon in the chat composer records via OpenAI Whisper. Useful when you're walking the auditor through what a project actually requires — easier to explain a year-long capstone aloud than to type it out. Recording stops on click; transcription replaces your in-progress text. You can edit the transcription before sending.

The readiness signal

On every turn, the auditor returns a structured readiness block that drives a progress strip below the chat. It shows:

  • Score, 0–100 — the auditor's honest estimate of how defensibly the Course Outcome Profile could be generated right now.
  • Covered — short labels for audit areas or competencies already locked in.
  • Remaining — what the auditor is still probing.
  • Good enough to generate — a boolean that flips to true at ~75.

The score is advisory, not gating. You can generate the profile at any score, and you can keep going past 90 if you have more to say. Most captures land in the 75–90 range; chasing 95+ usually means the auditor is asking refinement questions that won't materially change the scores.

When to stop

Stop when the auditor's questions are no longer surfacing things you'd care to defend at a curriculum meeting. If the next question is about a marginal sub-competency or a rubric line you'd score the same way regardless of how the conversation went, you have what you need.

§ 7CourseCapture · Generate & confirm the profile

When you're ready, click Generate Course Outcome Profile in the chat header. The system reads the full audit transcript plus all the materials and produces a structured profile in one pass. This is a separate AI call from the chat — it's the scoring step, not more conversation.

What you get

The profile renders below the chat as a structured editor with several sections:

  • Competencies (5–15 technical + 5 baseline foundational + any others evidenced). Each row is a one-sentence statement of what the course is evidenced to require, assess, and support, with K/U/D depth values (0–5 per dimension), evidence excerpts (verbatim quotes from materials or the transcript), and a short rationale. Foundational competencies (Agency, Attention to Detail, Resilience, Curiosity, Communication) score on Do only — K and U are null, not zero. A foundational with D=0 means the course doesn't create the conditions that develop the disposition, which is a valid and informative finding.
  • Incoming expectations (0–10). Skills the course assumes students arrive already able to do, with expected K/U/D depth and which assignments evidence the expectation. The downstream side of this — what later courses expect — feeds Explore's downstream-target mode.
  • Verification summary. A short fidelity-check block: course shape, strongest evidence, dimensional patterns (where K/U/D diverge meaningfully), catalog-vs-evidence findings, and a one-sentence foundationals glance. Read this section first — it's the "yes, that's my course" or "no, the system missed something" surface.
  • Audit notes. Prerequisite gaps, objective misalignments, cross-source conflicts, and (if any) suggested objective revisions, each as a one-sentence finding.
  • Revised objectives draft (optional). Only present if you explicitly asked for one during the chat. Informational only — it doesn't write back to the catalog.

Review the editor

Every depth value is editable: sliders adjust K, U, or D per competency, with evidence excerpts visible next to each cell. If a depth value looks wrong, edit it directly and note why in the rationale field. The audit notes section accepts edits too — keep what's useful, discard what isn't.

Closing prerequisite gaps back into the catalog

Each surfaced prerequisite gap in the audit notes carries a merge into skills button. Click it and the AI decomposes the gap into KUD+-tagged competencies, merges them with the course's existing Skills/Competencies Required list, and shows the unified replacement list inline (with new and clarified items color-coded). Hit copy merged list and paste it back into your course's Skills/Competencies Required cell on the shared sheet — that closes the loop between what the audit found missing and what the catalog says students need to arrive with. Re-clicking the button re-runs the merge if you've reworded the gap in the meantime.

Save Snapshot

Click Save Snapshot when the profile is faithful enough to publish. This writes an immutable, versioned record to the database, timestamped and tagged with the AI model that produced it. Snapshots are the historical record; the draft profile remains mutable for further edits. You can save multiple snapshots over time — each snapshot represents a confirmed view of the course at that point.

Snapshots vs. drafts

A snapshot is the artifact downstream views consume (the Program Coverage Matrix, Explore alignment, prereq-gap analysis if/when those views ship). A draft is your workspace — edit it freely, but it doesn't feed any downstream view until you save it as a snapshot. The draft persists between sessions; you don't lose work by closing the tab.

§ 8CourseCapture · Snapshots & revisions

Multiple snapshots over time

You can re-capture the same course as many times as you like. Each Save Snapshot creates a new versioned record; older snapshots are not overwritten. This is the unit of historical comparison: a "spring 2026 capture" and a "fall 2026 capture" of the same course can both exist, with their differences visible to anyone reading the snapshot list.

The Snapshots panel

The Snapshots panel on the capture page lists every saved snapshot for the course, newest first, with its caption, date, and the AI model that generated it. Click any row to view that snapshot's full profile in read-only form, including the verification summary and audit notes. Snapshots can be soft-retired (hidden from downstream views) but not edited or deleted in the prototype — they're a permanent record.

When to re-capture

Re-capture when something material has changed: a new project added, a major rubric overhauled, the prerequisites revised, the syllabus restructured. Cosmetic changes (a tightened sentence in the description) don't warrant a new snapshot. The Program Coverage Matrix automatically rescores against the newest snapshot per course, so a re-capture propagates into program-level views without further action.

Re-importing from Canvas before a re-capture is safe — file attachments upsert by (course, fileName), so existing rows are updated in place rather than duplicated. You don't need to clear old materials before clicking Import from Canvas again.

§ 9CourseCapture · Stress-testing the profile before you approve

After the profile has been generated but before you save a snapshot, you can run an adversarial review of it. The Stress-test this profile button appears on the review panel. Click it and a separate, heavier-tier AI agent reads the produced profile alongside the full audit transcript and all materials, then returns a structured critique — not an editorial rewrite, but a targeted search for places where the profile's claims outrun its evidence.

What the reviewer looks at

The reviewer works at two levels:

  • Per-competency annotations. For each competency, it returns a confidence indicator, any specific concerns (for example: a D=4 score where the only evidence is a single low-stakes exercise), and — optionally — a suggested score adjustment with a one-sentence rationale. It only annotates rows where it found something worth flagging; competencies it considers solidly evidenced show nothing.
  • Profile-level concerns. Three categories: catalog-vs-evidence mismatches (the catalog description implies something the materials don't support, or vice versa), internal consistency (K/U/D patterns within or across competencies that look contradictory), and coverage gaps (things the materials clearly develop that the profile doesn't capture). Each profile-level concern is a short finding with an explanation.

The reviewer also returns an overall assessment — a one-sentence judgment of whether the profile is ready to publish, needs minor refinement, or has substantive problems that warrant another audit conversation.

How the results appear

Competency rows that have concerns get an inline colored badge next to the row — the badge shows the concern count and opens the annotation on click. Rows the reviewer found acceptable show nothing. Profile-level concerns appear as a collapsible list below the competency table. The overall assessment appears at the top of the review panel.

The results are ephemeral: they're shown for this session only and are cleared the moment you make any edit to the profile. They don't persist into the snapshot. If you edit a depth value in response to a suggestion and want a fresh read on the revised profile, click the button again.

Cost and cadence

Each stress-test call costs roughly $0.05–$0.20, depending on course size. It's an on-demand tool — run it once when the profile looks mostly done, read the concerns, make adjustments you agree with, and ignore the rest. You don't need to run it to convergence; your judgment about the course is the ground truth the reviewer is challenging, not replacing.

The AI's job is to find problems

The reviewer is designed to be skeptical. Its concerns will sometimes be wrong — it may not know your course's context the way you do. Treat each concern as a question worth considering, not an authoritative verdict. If you read a suggestion and disagree, ignore it and move on. If a concern identifies something genuine you hadn't noticed, adjust the profile before saving. Either outcome is a good use of the tool.

§ 10Explore · Pick a snapshot and a mode

Open /explore/<course-code> — the same course code as in CourseCapture. The header shows a snapshot picker (defaults to the newest) and a mode toggle.

Snapshot picker

Pick which snapshot to analyze. If you have multiple snapshots for a course, this lets you run Explore against any of them — useful for asking "did our spring revision actually close the gaps Explore surfaced in fall?" Most of the time you'll work with the newest.

Two modes

ModeWhat it compares againstWhen to use
Custom targetA target you author in plain language — desired goals, capabilities, depths the course should reach. The AI structures your prose into a typed target spec.Course-revision conversations. "Should this course develop X to depth Y?" — answer that by writing the target and seeing where the current snapshot falls short.
Downstream targetThe union of incoming_expectations from every captured downstream course (courses that list this one as a prereq). Auto-detected; no authoring required.Prereq-sufficiency conversations. "Are we producing what the next courses in the sequence need?" — answer by running this mode and reading the gaps.

A session runs in one mode at a time. Combined-mode comparisons (custom and downstream in one view) are a future enhancement. Career-path alignment is intentionally deferred until graduate-outcome data is wired in.

§ 11Explore · Authoring a custom target

1
Pick the mode. Click Custom target in the mode toggle.
2
Type your goals in prose. One paragraph is fine; a bulleted list is fine. Describe what you want this course to develop, at what depth, with whatever framing comes naturally — you don't need to write in KUD+ format. Examples: "I want students to leave this course able to operate a press independently on standard substrates, and to articulate why each press setting matters." Or: "By the end of this course, students should be at a Do-4 level on color management for packaging work, and should have developed enough resilience to handle real client revisions."
3
Generate the target. Click Generate target spec. The AI proposes a structured target — a list of target competencies, each with K/U/D depth values, derived from your prose. You see it side-by-side with what you wrote.
4
Amend, then confirm. Edit the proposed target inline if it missed anything or interpreted you differently than intended. A short caption (optional) lets you label this target — "Spring 2026 redesign target" — so you can come back to it later. Click Save target to persist it.
5
Run the analysis. Click Generate analysis. The AI compares the snapshot to the saved target and produces the alignment output described in §13.
Multiple targets per course

You can save multiple custom targets attached to the same course — e.g., one for an aspirational long-term redesign and one for the more immediate next-revision target. Each shows up in the target list with its caption and date; running an analysis picks which target to analyze against.

§ 12Explore · Downstream target auto-detection

This mode works when at least one downstream course — a course that lists this one as a prereq — has itself been captured. Click Downstream target; the system queries the catalog for downstream courses, checks which have snapshots, and proposes a target derived from the union of their incoming_expectations.

What the proposed target looks like

Each downstream course's incoming expectations describe what it assumes students arrive able to do, at what K/U/D depth, with which of its assignments evidence the assumption. The union is what your course collectively needs to produce — if it serves as a prereq to GC 3400 and GC 4060, and those two have captured profiles, the target is the merged set of their incoming demands.

What if no downstream course is captured yet?

The mode tells you which downstream courses exist and which are missing snapshots. If none have been captured, the mode can't run — capture at least one downstream course first, or use the custom-target mode instead.

The chicken-and-egg of downstream mode

Downstream mode gets more useful as more courses are captured. Early in the program's adoption, your downstream courses might not have snapshots yet, so this mode shows partial coverage. As the captured-course list grows, the auto-detected target becomes richer — and the mode becomes the load-bearing one for prereq-sufficiency conversations.

§ 13Explore · Reading the analysis output

The analysis renders below the target with three sections.

Alignment table

One row per target competency, with how the current snapshot covers it. Status uses a four-level taxonomy:

StatusWhat it means
✓ coveredThe snapshot develops this competency at or above the target's depth on every dimension the target specifies. No action needed.
◐ partialThe snapshot addresses this competency but at lower depth than the target, or covers only some of the target's dimensions. Some closing work needed.
↓ underdevelopedThe snapshot touches the competency but the depth is so much shallower than the target that the course as currently structured won't reach it without substantive change.
✕ missingThe snapshot has no competency that addresses this target item. The course doesn't develop it at all.

Each row shows the target's required K/U/D, the snapshot's actual K/U/D, the matched snapshot competency (or "no match"), and a short rationale explaining the call.

Recommendations (2–4)

Ranked by impact on the most consequential gaps. Each recommendation is a one-sentence proposed change to the course — a new assignment, a depth increase on an existing competency, a structural shift in how a project is sequenced — followed by which target items it would address and an effort estimate. Recommendations are surfaced for instructor deliberation; they don't modify anything.

Audit notes

Three lists: gaps addressed by recommendations, gaps not addressed (the recommendations are best-effort but won't close everything), and strengths vs. target (where the snapshot is meeting or exceeding the target, worth preserving in any revision).

Analyses persist. Each saved analysis appears in the analyses list with its date, target, and snapshot — so you can revisit a months-old reading without re-running it, and so the historical thread of "we explored X against Y on date Z" is visible.

§ 14Explore · What-if scenarios

What-if scenarios let you propose a change to the course without actually re-capturing it, and see how the analysis would shift. Useful for course-revision deliberation: "if we added project X with depth Y on competency Z, would it close the gap on target item W?"

How to run one

1
Open an existing analysis. What-if scenarios attach to an analysis — pick the one whose target and snapshot you want to vary against.
2
Describe the proposed change in prose. "Add a 4-week packaging color project, scored on a substrate-management rubric, lifting Color Management to D=4." Or: "Drop the lecture-heavy first three weeks and replace with a press lab, shifting Resilience from D=1 to D=3."
3
Generate the scenario. The AI re-runs the alignment as if the proposed change were in the snapshot, and rates the change.

The worth_doing rating

Each what-if result includes a categorical judgment:

  • worth doing — the change closes consequential gaps and the effort is proportionate.
  • worth considering — the change has merit but trade-offs or effort are non-trivial; faculty deliberation warranted.
  • not worth doing — the change doesn't materially shift the alignment, or addresses a peripheral gap at high cost.

Plus a short rationale and the specific target items the scenario would address vs. fail to address.

What-if is speculative

The scenario re-rates against a hypothetical snapshot you described in prose, not against actual student work or actual assignment evidence. It's a thinking aid, not a substitute for re-capturing the course after the change has actually been implemented. Use it to filter candidate changes before doing the work; use a fresh capture to confirm the change had the effect you wanted.

§ 15The /courses roster

The Course Catalog at /courses is the program-wide control panel for capture status and prerequisite work. It lists all 120 GC catalog courses grouped by level (1000 / 2000 / 3000 / 4000), with a data-state badge per row and controls for bulk-loading courses and adding new entries.

Data-state badges

Every row shows one of three badges:

BadgeWhat it means
capturedThe course has at least one confirmed, non-retired snapshot. The prerequisite-gap view is available at /courses/[code].
in-auditA capture session is in progress (audit messages exist in the last 24 hours) but no snapshot has been confirmed yet.
not-startedNo capture session has begun. The course exists in the catalog but has no materials or audit history.

Bulk preload

The Preload all button at the top of the roster runs a bulk sync of catalog data (from the shared Google Sheet) for every course in the list in one pass, rather than requiring you to visit each course's Materials panel individually. Use this at the start of a new semester to ensure the catalog row for every course is current before captures begin. The button shows a progress indicator as it processes courses in batches; it can be stopped mid-run.

Add a class

If a course exists in the program but is not yet in the catalog (a new course number, a new special-topics section), use the Add a class form at the bottom of the roster. Enter the course code, title, level, and a short description. The course row is created immediately; you can then open /capture/[code] and begin a capture session as with any existing course. Coordinate with the administrator before adding courses — the roster is shared across all faculty using the tool.

§ 16Prerequisite confirm & gap view

Click any captured course row in the roster to open /courses/[code], the per-course prerequisite surface. This is where you seed the skill-tagged prerequisite edges for the course, confirm or edit them, and read the computed prerequisite-gap view.

Step 1 — Seed edges from the syllabus prerequisites

The course's catalog entry lists its declared prerequisites as free text (e.g., "GC 2150 and GC 2800 or consent of instructor"). The Seed prerequisite edges button runs the prereq-edge-seed AI function, which reads that free text alongside the course's incoming_expectations from any existing capture and proposes a structured list of edges: for each prerequisite course code it identifies, it names the specific sub-competency it believes the focal course's assignments depend on, along with the expected K/U/D depths. One pass is usually sufficient; run it again if you add materials or re-run the audit.

Step 2 — Confirm or edit the edges

The proposed edges appear in an editable list. For each row:

  • Prerequisite course code — the upstream course the edge links to. If the seeder misread the free-text prereqs (course codes can be ambiguous), correct this here.
  • Sub-competency — the specific program sub-competency this edge is tagged to. You can change this if the seeder chose a different sub-competency than you'd pick.
  • Expected K / U / D — the depth this course's assignments presuppose from incoming students. Adjust if the seeded values don't match what you'd actually grade students on.

Click Confirm edges to persist the list. You can re-seed at any time to regenerate the proposals, or edit individual rows by hand at any point.

Step 3 — Read the gap view

Once edges are confirmed, the gap view renders below the edge list. For each confirmed edge, it shows:

ColumnWhat it means
Prerequisite courseThe upstream course the edge links to.
Sub-competencyThe specific skill the gap is about.
Expected K / U / DWhat this course's assignments presuppose.
Attained K / U / DThe depth the prerequisite course actually reached, per its confirmed snapshot (or intended estimate).
Gap (per dimension)Expected minus attained, per dimension. Positive = gap; zero or negative = met or exceeded.
BasisHow the attained depth was determined — see below.

What "syllabus-promise" means

The basis field distinguishes the quality of evidence behind the attained depth:

  • measured — the prerequisite course has a confirmed Course Outcome Profile. The attained depth comes from a real capture backed by assignments and instructor confirmation.
  • syllabus-promise (also labeled intended in the tool) — the prerequisite course has not been captured, but a depth estimate was derived from its catalog description or your course's incoming_expectations field. This is what the course claims to develop, not what it's been confirmed to produce. It is marked as unverified; treat it as a placeholder pending a real capture of that course.

What "not yet captured" means

When the basis reads not yet captured (displayed as a no_data state in the tool), the prerequisite course has no snapshot and no intended estimate was available. The gap engine explicitly returns no data for that edge — it does not assume the gap is zero (which would hide a real problem) or assume the gap is maximal (which would generate false alarms). The only resolution is to capture the prerequisite course. Until then, the gap for that edge is genuinely unknown.

No double-counting across multiple prereqs

If two prerequisite courses both develop the same sub-competency, the gap engine takes the maximum attained depth across the two, not the sum. This is intentional: KUD+ depth values are ordinal levels (not additive quantities), and a student who practiced a skill in two courses at depth 3 each has attained depth 3 — not depth 6. The maximum reflects the best preparation available from any prerequisite; supplementary coverage adds practice value but doesn't change the attained level for gap purposes.

Most useful once prereqs are captured

Like Explore's downstream mode, the gap view gets richer as more courses are captured. Early on, many edges will show "not yet captured" for the upstream course. As the captured-course list grows, the basis improves from "not yet captured" → "syllabus-promise" → "measured," and the gap judgments become increasingly reliable.

§ 17Tips & gotchas

For CourseCapture

  • Load Canvas before talking to the auditor. The auditor's opening turn is much sharper when it can see actual assignment text and point values. Doing a Canvas import after a few audit turns sometimes triggers the auditor to revise earlier findings, which can be confusing.
  • Trust the readiness signal — but verify the verification summary. A score of 80+ usually means the profile will be solid, but the verification summary is the last fidelity check. Read it before clicking Save Snapshot.
  • Don't pad foundational competencies. A foundational with D=0 is a real finding — it means the course doesn't develop that disposition. Inflating it to D=2 because "every course teaches communication a little" undercuts the signal. The point of scoring is to surface where dispositions actually live in the program.
  • Inaccessible content gets flagged conservatively. When the auditor sees a YouTube link or Canvas Page it can't read, it surfaces the reference and asks what's in it. If you don't tell it, the score stays low — the auditor explicitly avoids over-scoring on inaccessible content.
  • Voice input is most useful for narrative explanations. When the auditor asks "what does the capstone actually require?" — easier to talk through than to type. For one-word answers or specific corrections, typing is faster.

For Explore

  • Custom-target mode is the higher-value mode early on. Downstream mode requires other courses to have been captured; custom mode is self-contained and drives most course-revision conversations.
  • Write your custom target in your own words. The AI structures the prose into KUD+ format; you don't need to think in K/U/D depths to author a target. Concrete examples (which assignment, what depth, why) produce better structured targets than abstract goal statements.
  • The recommendations are a starting point. They're 2–4 ranked, evidence-aware suggestions — not a complete redesign. Use them as the agenda for a faculty conversation, not as the answer.
  • What-if scenarios are cheap. Run multiple scenarios per analysis to compare candidate changes. The cost per scenario is small (the model runs are quick); the value is in filtering the space of possible changes.

§ 18Sharing the read-only view

Every captured course has a public profile page at /view/<course-code>. This address requires no authentication and works over plain HTTP on the LAN — anyone who can reach http://130.127.162.180:3000/ can open it. The page shows the course's captured essence, outcomes, and the "how it differs from the catalog" notes when a snapshot exists; if the course hasn't been captured yet, it falls back to the catalog description.

What the read-only view is for

Share /view/<course-code> links with people who need to read what a course is actually evidenced to do but shouldn't be in the edit flow:

  • Department chairs and associate deans reviewing the curriculum ahead of an accreditation visit. They can open every course's profile without needing a slug or credentials.
  • Accreditation reviewers. A link to the public landing (http://130.127.162.180:3000/) gives auditors a live, browsable view of every captured course with its status. The catalog-vs-evidence notes in each profile speak directly to the kind of currency and alignment evidence accreditors typically request.
  • Students researching what a course actually develops, beyond the catalog blurb.
  • Colleagues at other institutions you'd like to share a profile with informally — point them to the view link; they don't need access to the full tool.

What the read-only view does not show

The view page shows the confirmed snapshot's outcomes and essence. It does not show the raw audit transcript, individual competency K/U/D scores, audit notes, or the review panel — those are internal working documents that live on the edit surface. The intent is a clean, faculty-authored statement of what the course does, not the full analytical machinery behind it.

Transparent curriculum framing

The public landing and read-only views embody a principle: the curriculum itself is public. Anyone can read what the GC program claims each course develops; only faculty can change the record. This gives chairs, accreditors, and students a reliable, up-to-date reference without creating an edit-access management problem. The Basic Auth gate on the edit surface protects the writing; nothing protects the reading.

§ 19What these tools don't do

  • Not student or faculty evaluation. Depth scores describe what the course's materials and assignments require, not how well any instructor delivers the course or how well any student performs in it.
  • Not automated curriculum redesign. Explore proposes recommendations; faculty deliberation and CourseCapture re-runs are how revisions actually happen.
  • Not a replacement for faculty judgment. The auditor drafts, the instructor confirms. Every depth value is editable; every recommendation is a starting point.
  • Not career-path alignment in Explore. Career destinations are addressed in the Program Coverage Matrix (live at /program) — Explore's career-path mode is deferred until graduate-outcome data is wired in.
  • Not a snapshot diff. You can read multiple snapshots side by side from the Snapshots panel, but the system doesn't yet produce a structured diff highlighting what changed between captures. That's a planned Phase 2 view.
  • Not extensible to video on non-YouTube platforms or external-tool content automatically. Vimeo, Panopto, Canvas Studio, Kaltura, Loom, and external LTI items are flagged as inaccessible references; the auditor will ask you what's in them, but the system can't read them itself in the current build. (Canvas Pages, file attachments, and quiz content on both Classic and New Quizzes are all read — see the source-coverage table in the Background document.)

For the underlying framework — what K/U/D depth means, why the scoring is structured this way, what counts as evidence — see the KUD+ Background document. For the program's overall direction and what's coming next, see the Vision document.
